# 8. Conclusion

This paper has shown that MQTT's topic hierarchy and subscription semantics provide a sufficient foundation for implementing a distributed database without modifying or extending the wire protocol. The topic-to-database mapping (Section 3) formalizes how the hierarchical topic space encodes entity names, record identifiers, and operation verbs, while two distinct MQTT mechanisms — request-response for synchronous CRUD and subscriptions for change data capture — cover the full spectrum of database interaction patterns. Atomicity is supplied by the storage engine's per-request batch commit, not by the wire protocol: a standard MQTT 5.0 client observes atomic CRUD by treating the Request/Response reply as the success signal, with no client-side transaction logic and no broker-specific protocol extension.

MQDB demonstrates this mapping in a concrete system: a single Rust binary that functions simultaneously as an MQTT 5.0 broker and a distributed document database. The architecture (Section 4) uses the same topic semantics for partition routing, asynchronous replication, and distributed constraint enforcement, with a Raft control plane for cluster membership and partition assignment.

The unified architecture eliminates the two-system problem that characterizes conventional IoT data infrastructure. Authentication is configured once, not twice. Access control covers both messaging and data operations in a single policy. There is no consistency gap between the broker and the database because they are the same process. And capabilities like protocol-level field encryption (Section 5) arise naturally from the unification — they are structurally impossible in separated architectures without additional middleware. The architectural-workload experiments in §6.4 operationalise this thesis by measuring three primitives — change-feed delivery, unique-constraint contention, and cascade delete — whose atomicity guarantees the separated baselines cannot match without introducing additional transactional infrastructure that reconstructs, piecemeal, what the unified architecture provides natively.

The mapping has explicit boundaries: no multi-record transactions, no cross-entity joins, no SQL-level query expressiveness, and QoS levels do not yet map to durability guarantees. These limitations are inherent to topic-based addressing and define the system's applicability to operational IoT data management rather than analytical workloads.

To our knowledge, this is the first academic treatment that formalizes the argument for MQTT as a database foundation, describes a complete distributed architecture built on this mapping, and evaluates the resulting system against conventional separated alternatives. The convergence of messaging and storage — observed independently in Kafka, EMQX, and Harper — is not an accident. It reflects the fact that the boundary between these systems, long assumed fundamental, is an artifact of historical design rather than a necessary architectural constraint.

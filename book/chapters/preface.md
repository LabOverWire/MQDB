# Preface

This book was supposed to be about building a database. It turned into a book about what happens when you pick the right constraint.

MQDB started with a narrow bet: use MQTT as the protocol for a distributed database. Not HTTP, not gRPC, not a custom binary RPC. MQTT — the protocol designed for IoT sensors and constrained devices. The reasoning was pragmatic. MQTT 5.0 had request-response support, persistent connections, and client libraries in every language including embedded C. If the database spoke MQTT, any device that could publish a message could also store and query data. No custom SDK required.

What we did not anticipate was how many features would fall out of that decision for free.

The first surprise was change events. In a traditional database, if you want consumers to react when data changes, you need Change Data Capture — a separate process that tails the write-ahead log and publishes events to a message broker. CDC requires its own infrastructure, its own failure modes, its own operational burden. In MQDB, the database already lives inside a pub/sub broker. When a record is created, the system publishes to `$DB/users/events/created`. Clients subscribe with standard MQTT wildcards — `$DB/users/events/#` catches all changes to the users entity. The feature cost almost nothing to build because the infrastructure already existed for a different reason.

The second surprise was ownership. MQTT authenticates clients at connection time. The broker knows who you are for the duration of your session. When a client publishes to `$DB/users/create`, the handler extracts the authenticated identity from an MQTT user property — `x-mqtt-sender` — and injects it as the record's owner. Read, update, and delete operations check that the requester matches the owner. No separate authorization service, no token validation middleware, no row-level security policy language. The MQTT connection already carries the identity. The database just reads it.

The third surprise was vault encryption. MQDB encrypts sensitive fields at rest using AES-256-GCM, with per-record keys derived from a user passphrase. The encryption happens transparently — clients send plaintext JSON, and the system encrypts before storage, decrypts before response. This works because every database operation passes through a single MQTT publish handler. That handler was designed to parse topics, extract payloads, and dispatch to the storage engine. Inserting an encryption step between "parse the request" and "write to storage" required modifying one function. The interception point was not designed for encryption. It was designed for MQTT message handling. Encryption just happened to fit.

The fourth surprise was distributed routing. In cluster mode, MQDB partitions data across nodes using a hash of the entity name and record ID. The topic `$DB/users/user-42` contains both the entity (`users`) and the ID (`user-42`) — exactly the inputs to the partition function. The MQTT topic structure, designed as a human-readable API, accidentally became the routing key for the distributed system. When a request arrives at the wrong node, the system parses the topic, computes the partition, and forwards to the primary. The topic is simultaneously the API endpoint and the shard key.

The fifth surprise was access control. MQTT brokers have topic-based ACLs — rules that control which clients can publish or subscribe to which topics. Because database operations map to MQTT topics (`$DB/users/create`, `$DB/orders/list`), the existing MQTT ACL infrastructure controls database access without modification. Grant a client publish access to `$DB/orders/#` and they can create, read, update, and delete orders. Revoke it and they cannot. Event streams are read-only topics (`$DB/users/events/#`) protected by a separate tier. Admin operations live under `$DB/_admin/#` with their own restriction. Three protection tiers, applied uniformly, using the same mechanism the broker already uses for regular MQTT topics.

None of these features were in the original design document. The design document said: build an MQTT broker with an embedded key-value store. Use MQTT 5.0 request-response for CRUD operations. Partition data for distribution. That was the plan.

What happened instead was that each new requirement — real-time events, row-level security, encryption, sharding, access control — turned out to be a small extension of something that already existed. The pub/sub engine gave us CDC. The authenticated connection gave us ownership. The publish handler gave us an encryption hook. The topic structure gave us a partition key. The ACL system gave us entity-level authorization.

This is not a story about clever engineering. It is a story about architectural constraints. MQTT imposed a specific shape on the system: everything is a message, every message has a topic, every connection has an identity, every topic has access rules. Those constraints turned out to be load-bearing. Features that would require dedicated infrastructure in a traditional architecture — a CDC pipeline, an authorization service, an encryption middleware, a routing layer — collapsed into properties that the protocol already provided.

The lesson generalizes beyond MQTT. When you choose the right foundational abstraction, features compound. When you choose the wrong one, they fight. The difference is not visible at the start. It reveals itself over months of implementation, as each new requirement either slots into the existing architecture or requires scaffolding to work around it. In MQDB, the ratio was roughly 5:1 — five features that fell out naturally for every one that required genuine engineering (distributed unique constraints, Raft consensus, partition migration, session handoff, cross-node query coordination). The hard problems were real. But the easy ones would not have been easy under a different protocol choice.

This book follows the construction of MQDB from storage engine to distributed cluster. Each chapter covers a component: what problem it solves, what design options were considered, how it was implemented, what went wrong, and what the experience teaches. The chapters are technical. They contain partition maps, replication protocols, consensus algorithms, wire formats, and failure detection state machines.

But underneath the technical narrative runs a second story: the story of an architecture that kept giving back more than was put in. Pay attention to the moments where a new feature requires surprisingly little new code. Those are the moments where the protocol constraint is doing the work. They are also the moments that reveal whether a system's foundation was well chosen — not by theory, but by the accumulating evidence of implementation.
